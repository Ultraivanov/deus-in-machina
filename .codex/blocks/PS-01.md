## Block Goal

Make the released package and runtime smoke paths observable and reproducible so post-release breakages are caught by explicit verification instead of slipping past unit tests.

## Definition of Done

A documented smoke suite runs against the shipped package surface and core runtime scripts, fails on the currently confirmed regressions, and produces a stable checklist of post-release failures to drive repair blocks.

### PS-01-T1 — Define release smoke matrix

**Files to modify:**
- `.codex/blocks/PS-01.md` — mark task in-progress and log plan

**Files to create:**
- `docs/release-smoke-matrix.json` — machine-readable smoke check manifest
- `docs/release-smoke-matrix.md` — human-readable smoke check documentation

**Files NOT touched:**
- `src/`
- `scripts/`
- `package.json`
- `README.md`
- fixtures and expected outputs

**Approach:**
Define a stable smoke-check contract first, before writing any runner code. Encode
the release checks as a small JSON matrix with ids, commands, expected exit behavior,
and purpose, then add a short Markdown companion that explains the checks and records
the currently confirmed regressions this matrix is meant to expose.

**Risks:**
If the matrix format is too ad hoc, `PS-01-T2` will have to reshape it. Keep it
minimal and command-oriented so the runner can consume it directly.

### PS-01-T2 — Add executable smoke runner

**Files to modify:**
- `.codex/blocks/PS-01.md` — mark task in-progress and log plan

**Files to create:**
- `scripts/run-release-smoke.js` — smoke runner that executes the matrix and reports structured results
- `docs/release-smoke-results.example.json` — example result payload for the runner

**Files NOT touched:**
- `src/`
- existing vector runners
- `package.json`
- `README.md`
- fixtures and expected outputs
- `docs/release-smoke-matrix.json`
- `docs/release-smoke-matrix.md`

**Approach:**
Implement a standalone smoke runner that consumes `docs/release-smoke-matrix.json`,
shells out to each declared command, and records per-check command, exit code,
pass/fail, stdout/stderr excerpts, and timing. Keep the task read-only with respect
to product code so it adds observability without starting repairs.

**Risks:**
Some smoke commands mutate `.actual.json` or `pipeline-run` artifacts today. The runner
should report that reality rather than hide it; cleanup can be handled separately if needed.

### PS-01-T3 — Capture baseline failures and triage output

**Files to modify:**
- `.codex/blocks/PS-01.md` — record completion and triage note

**Files to create:**
- `docs/release-smoke-results.baseline.json` — baseline smoke results captured from current repository state
- `docs/release-smoke-triage.md` — failure analysis and mapping to repair blocks

**Files NOT touched:**
- `src/`
- `scripts/` implementation files
- `package.json`
- `README.md`
- fixture expectations

**Approach:**
Run the smoke runner against the current repository state, save the structured baseline
results, then document each failing or suspicious check and map it to the repair block
that should handle it next.

**Risks:**
The smoke run mutates `.actual.json` and `pipeline-run` artifacts, so those generated
changes must be restored after capturing the baseline report.
